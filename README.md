# graphADT

This project's goal was to create an implementation of a graph abstract data type and perform several different operations on graphs that we create. I used the library JgrapT as my graphADT that I could use to create and manipulate the graphs, as well as run algorithms on in order to check graph properties. 

## Problems to Solve  
1.  Create undirected graphs of sizes of 10, 100, 10000 nodes with the number of edges being 80% of nC2. Example: If the graph has 10 nodes, then the number of edges should be 45 and every node should have at least an edge. The edges can be randomly connected to the nodes. Test the graph that you are created such that every node has an edge. 
2.	Compute the strongly connected components of the graphs that you have created in (1) and record the time for the algorithm to compute it. 
3.	Enumerate the nodes of the graph created in (1) with numbers. Example, take a node, a labeled it with 1 and the node that a has edge with, label it 2.  Create bipartite graph where there is a directed edge from an even numbered node to an odd numbered node and vice versa. 

## Report  
### Create and Test Undirected Graphs
I first started by creating a function that supported the creation of undirected graphs of size 10, 100, and 10,000. When I tried to create a graph of size 10,000 everything was ok, but when trying to add the 39,996,000 edges (80% of 10,000 choose 2), I ran out of memory on my heap even after trying to allocate more memory. After many attempts to fix the problem I accepted a small defeat and moved on with only creating graphs of size 10, 100, and 1,000. I created a function to add all of the necessary edges to each graph by creating a while loop that continued looping for (n choose 2) * 0.8 times to satisfy a condition in the homework problem. Before creating all of the edges I started with connecting each vertex to each other to guarantee that all vertices had at least one edge. I decided to create a random edge by finding two random integers from size 0 to the number of vertices using Math.random(). After getting the random numbers I created an edge between the two corresponding vertices and checked to see if it already existed. If it didn't already exist, I incremented the counter and continued into the while loop was finished.  When testing to see if every vertex had an edge I just went through each vertex and checked for at least one edge. I added a function to return true or false depending on if every node had an edge.  

### Compute strongly connects components  
In order to calculate the strongly connected components I used a function that implemented Kosaraju's algorithm for finding strongly connected components. The algorithm works by performing two depth first search (DFS) passes. The first pass, once the DFS algorithm is called on all adjacent nodes, the main node is put on the stack. After this step, the algorithm reverses the graph, or transposes, it. In the final step we pop a vertex from the stack and perform a DFS on each one of the nodes and the DFS of each node prints the strongly connected component of the node popped off the stack. In my case, I had one large strongly connected component for each of my graphs every time. This is because of the way I create the graph. I start by connecting each node to the one after it to guarantee each node has an edge. Because of this, I end up with one large strongly connected component for each of my graphs. When calculating the time, it takes for each graph's strongly connected component to be calculated, I found that the first two graphs of size 10 and 100 each around 5 to 10 milliseconds while the third graph of size 1000 takes roughly 200 plus milliseconds each time. This is to be expected since not only is the number of vertices growing by 900, the edges are growing by 39,992,040 edges. Since this is the case, such a large time difference is to be expected.  

### Create a Bipartite Graph From Existing Graph  
This problem seems very straight forward because there is an explanation on how to do it in the homework. However, I went in assuming that a bipartite graph would have the same amount of edges as n choose 2 * 0.8 as it was above. This is not the case. When implementing the solution, I started with a test that could check if a graph was indeed bipartite and would return true or false depending on the results. I set a conditional in the while loop that checked to see if an edge already existed, and using the same random numbers as before, I added them together and checked to see if random number 1 plus random number 2 was not divisible by 2. This is a rule in mathematics that an odd number plus an even number will always be odd. An even number plus and even number will always be even. An odd number plus an odd number will always be even. I was able to use this to my advantage since I only wanted even nodes connected with odd nodes, as long as my condition was met, I would add the edge. I ran into my first problem when I tried to add the edges and my checkBipartite function kept returning false. After looking into the issue, it seemed that since we were working with directed graphs, all even had to point to odd or all odd had to point to even. There could not be a mix of some odd pointing to even and some even pointing to odd. I solved this by adding another conditional that checked to see if the first of the two numbers were even to guarantee that the edge would always go from even to odd. After solving this, I found that when creating the first graph I kept having my program run infinitely after it had created 25 edges. I went into debugging the problem for some time and discovered that the size 10 graph would freeze after adding 25 edges, the size 100 graph would freeze after adding 2500 edges, and the size 1000 graph would freeze after adding 25000 edges. I got my pencil and paper and solved it out using algebra. I found that if added 10^2 * 0.25 edges, everything worked great. The function to check if it was bipartite would return true every time. I later looked up to discover the maximum amount of edges in a bipartite graph is 25% of V^2 where V is the number of vertices. I felt proud that I had reached this conclusion on my own before consulting outside resources.
